# Workflow Handbook — Memory and Learning Loop

## Purpose
Define the process for maintaining structured project memory, logging lessons, writing retrospectives, and feeding improvements back into team handbooks.

## Overview
The memory and learning loop is a continuous improvement process:
1. **Log** lessons as they occur during runs and reviews
2. **Retrospect** after completed runs to capture what worked and what didn't
3. **Summarize** the current project state in a durable memory artifact
4. **Check** memory coverage and staleness periodically
5. **Revise** handbooks when lessons reveal recurring patterns

## Artifacts

### Project Memory (`memory/PROJECT_MEMORY.md`)
A durable summary of the project's current state — milestones, scripts, handoffs, outputs, handbooks, and memory entries. Auto-generated by scanning the repo; not hand-written.

**When to refresh**: after milestone completions, structural changes, or whenever a fresh session needs a recap.

**Command**:
```bash
docker compose -f docker/docker-compose.yml run --rm gis-worker \
  python scripts/update_project_memory.py
```

### Lessons Learned (`memory/lessons-learned.jsonl`)
An append-only JSONL log of lessons with structured fields:
- `timestamp` — when the lesson was logged
- `category` — broad category (data-quality, pipeline, tooling, process, qa)
- `tags` — specific tags for filtering
- `lesson` — one-line description
- `run_id` — (optional) which run this relates to
- `detail` — (optional) longer explanation

**When to log**: after any run, review, incident, or discovery worth remembering.

**Command**:
```bash
docker compose -f docker/docker-compose.yml run --rm gis-worker \
  python scripts/log_lesson.py "The lesson text" \
  --category data-quality --tags retrieval census --run-id my-run
```

### Retrospectives (`memory/retrospectives/<run-id>.retro.md`)
Structured markdown retrospectives for completed runs or milestone chains. Include:
- pipeline summary (which stages ran, their status)
- what worked
- what could be improved
- suggested improvements

**When to write**: after every completed pipeline run, especially runs with warnings or failures.

**Command**:
```bash
docker compose -f docker/docker-compose.yml run --rm gis-worker \
  python scripts/write_retrospective.py my-run-id "My Run Title" \
  --provenance runs/provenance.json \
  --processing-handoff runs/processing.json \
  --analysis-handoff runs/analysis.json \
  --validation-handoff runs/validation.json \
  --reporting-handoff runs/reporting.json \
  --lead-handoff runs/lead.json
```

### Memory Status
A health check that scans for missing or stale memory artifacts.

**Command**:
```bash
docker compose -f docker/docker-compose.yml run --rm gis-worker \
  python scripts/check_memory_status.py
```

## Handbook Revision Process

When lessons accumulate around a specific topic, revise the relevant handbook:

1. Review lessons filtered by category/tags
2. Identify recurring patterns or gaps
3. Edit the relevant handbook (role, workflow, source, or QA)
4. Log a lesson noting the revision: `"Updated <handbook> based on lessons about <topic>"`
5. Refresh project memory

This is a manual process by design — handbook changes should be deliberate, not automated.

## Cadence

| Trigger | Action |
|---|---|
| Run completes | Log lessons, write retrospective |
| Milestone completes | Update project memory, write retrospective |
| Structural change (new scripts, handbooks) | Refresh project memory |
| Weekly check-in | Run `check_memory_status.py` |
| 3+ lessons on same topic | Consider handbook revision |

## Portability

The memory layer is file-based and self-contained within the project directory. To deploy on a fresh instance:

1. Clone or copy the project repo
2. Run `python scripts/check_memory_status.py` to see current state
3. Read `memory/PROJECT_MEMORY.md` for a quick project recap
4. Read `memory/lessons-learned.jsonl` for accumulated knowledge
5. Read retrospectives in `memory/retrospectives/` for run-level history

No external services or databases required.
